# 모드 프로토콜 (SINGLE / MULTI / METAPHOR)

---

## SINGLE 모드 — 하나 깊게

**트리거:** 1개 대상, "깊게 분석해줘", "정밀 벤치"

**프로토콜:**
```
1. 스코프 판정 (A/B/C/D)
2. B1~B5 5블록 순서대로 생성
3. 각 블록 3~6줄 상한
4. 백본 수렴 + 확신도
5. 총 분량: A4 반쪽 (1000~1500자)
```

**필수:** B4 차용포인트 1~3개. "그래서 우리가 뭘"이 없으면 SINGLE 실패.

---

## MULTI 모드 — 주요 N개 비교

**트리거:** 2~5개 대상 동시, "비교해줘", "A vs B"

**프로토콜:**
```
1. 각 대상 스코프 동일한지 확인 (다르면 정렬 축 필요)
2. 대상별 5블록 압축 작성 (블록당 2~3줄로 더 짧게)
3. 비교축 표 생성 (최소 3축)
4. 통합 차용포인트 — N개에서 뽑은 공통+상충 패턴
5. 공통 백본 1줄 or 대조 백본 2줄
```

**비교축 표 예시** (미디어 비교: HBR·Substack·Medium)

| 축 | HBR | Substack | Medium |
|---|---|---|---|
| BM 핵심 | B2B교육 | 크리에이터 구독분할 | 멤버십+광고 |
| 콘텐츠 | 연구기반·에디팅 | 개인·이메일 | 중간층 |
| 락인 | 브랜드 권위 | 이메일 리스트 | 추천 알고리즘 |

**통합 차용포인트:**
```
- 세 모델 공통: "콘텐츠 → 구독"
- 차별화 축: 권위(HBR) vs 개인(Substack) vs 알고리즘(Medium)
- 우리 기획에 적용: 권위+이메일 조합 (HBR+Substack 하이브리드)
```

**금지:** 비교축 없이 5블록 단순 나열 → MULTI 실패.

---

## METAPHOR 모드 — 타 도메인 은유

**트리거:** "~처럼", "~에 적용", 원 도메인에서 막힘

**프로토콜:**
```
1. 원 도메인 대상 SINGLE (5블록)
2. 핵심 메커니즘 1~2개 추출 ("써드 플레이스", "리추얼")
3. 타겟 도메인에 매핑
4. 충돌지점 2~3개 명시
5. 변형안 (충돌을 어떻게 우회)
6. 차용 가능 / 불가 판정
```

**예시 구조:**
```
[원] 스타벅스 (5블록 요약)
     → 핵심 메커니즘: "써드 플레이스 + 리추얼 + 브랜드 일관성"

[매핑] 은행 지점 벤치
     - 써드 플레이스 → 업무+휴게 공간화
     - 리추얼 → 방문 체크인·개인맞춤 인사
     - 일관성 → 전 지점 동일 감각

[충돌]
     - 금융은 프라이버시·속도 중요 vs 스벅은 체류·개방
     - 은행은 지점 방문 감소 추세 vs 스벅은 방문 필수

[변형]
     - 디지털: 속도·프라이버시 우선
     - 오프라인 플래그십 소수만 써드 플레이스화
     - "가끔 가는 의식 공간"으로 포지셔닝

[판정]
     - 차용 가능: 리추얼·일관성 (전 채널)
     - 차용 불가: 공간 자체 (규제·동선 다름)
     - 확신도: 보통(70) / 근거: JPMorgan·KB의 플래그십 지점 실험
```

**은유 카탈로그:** `→ 04_metaphor_library.md`

## 모드 간 복합

| 요청 | 권장 |
|---|---|
| "HBR·Substack·Medium 비교 + 은유" | MULTI → 메타퍼는 별도 섹션 |
| "HBR 깊게 + 우리와 차이" | SINGLE → B4에 차이점 명시 |
| "스타벅스처럼 은행 만들려는데 참고사례" | METAPHOR + MULTI (스벅 외 쓰리플 사례) |

## Gotchas

| 함정 | 대응 |
|---|---|
| SINGLE인데 블록별 2~3줄로 너무 얕게 | SINGLE은 3~6줄. MULTI가 2~3줄 |
| MULTI 비교축에 "좋다/나쁘다" | 축은 구조·메커니즘. 가치판단 금지 |
| METAPHOR 충돌지점 생략 | 매핑만 하고 충돌 안 보면 오용 발생. 필수 |
| MULTI N>5 | 5개 상한. 초과 시 카테고리 나눠 MULTI 2회 |
