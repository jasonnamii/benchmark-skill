# Benchmark Skill — 짧고 맥락 있는 벤치마킹 엔진

대상을 5블록(정의+성공실패 / 구조 / WHY / 차용포인트 / 백본1줄)으로 압축. 기획에 즉시 투입 가능한 "맥락 살아있는 사례".

**용어 동의어** — 벤치마킹 = 사례분석 = 레퍼런스분석 = 경쟁사분석 (혼용). 성공사례·실패사례 대조는 B1 블록이 담당. 타도메인 차용은 **은유벤치마킹(크로스벤치)** = METAPHOR 모드. benchmarking 영문 트리거 포함. 벤치마킹스킬=본 스킬.

---

## ⛔ 절대 규칙

| # | 규칙 | 이유 |
|---|------|------|
| 1 | **HBR 밀도 준수** — 1개 대상 A4 반쪽 이내. 백본 1줄 필수. | 긴 벤치는 읽히지 않음. 압축이 생명 |
| 2 | **5블록 구조 고정** — 정의+성공실패 → 구조 → WHY → 차용포인트 → 백본 | 블록 빠지면 기획 투입 불가 |
| 3 | **확신도 병기 + 근거 1줄** — 높음(90)/보통(70)/낮음(50)/모름(30) | 환각·추측 차단 (UP v37) |
| 4 | **근거 없으면 "모름" 선언** — 패턴완성 금지 | 비공개 매출 추정하지 말 것 |
| 5 | **hedge 9종 금지** — "수 있·듯·가능성·것 같" 등 단정형 | UP v37 독립성 규칙 |
| 6 | **차용포인트 필수** — "그래서 우리가 뭘 할지" 1줄 이상 | 벤치의 존재 이유 |

## 실행 흐름

```
① 입력 수신 → ② 스코프·모드 판정 → ③ 5블록 생성 → ④ 백본 수렴 → ⑤ .md 산출
```

### ① 입력 수신

| 입력 유형 | 예시 | 다음 |
|---|---|---|
| 1개 대상 | "HBR 벤치마킹해줘" | SINGLE |
| 2~5개 대상 | "HBR·Substack·Medium 비교" | MULTI |
| 타도메인 은유 요청 | "은행을 스타벅스처럼" | METAPHOR |
| 대상 불명확 | "미디어 사례" | 범위 확인 질문(AskUserQuestion) |

### ② 스코프·모드 판정

**스코프 4분법** — 대상 성격으로 자동 분류. 블록 구조가 스코프별로 살짝 변형.

| 스코프 | 구조 블록 변형 | 예시 |
|---|---|---|
| A. 조직·서비스·BM | 조직/서비스스택/BM/규모 | HBR, Stripe, 쿠팡 |
| B. 제품·기능·UX | 핵심기능/UX패턴/기술스택/성장레버 | 노션, 토스, Duolingo |
| C. 캠페인·마케팅·카피 | 메시지/채널/크리에이티브/결과지표 | 애플 1984, 나이키 Just Do It |
| D. 정책·제도·사회현상 | 배경/제도설계/이해관계자/결과 | 싱가포르 HDB, 덴마크 플렉시큐리티 |

상세: `→ references/01_scope_blocks.md`

**모드 프로토콜:**

| 모드 | 트리거 | 산출 |
|---|---|---|
| SINGLE | 1개 대상 | 5블록 × 1 + 백본1줄 |
| MULTI | 2~5개 대상 | 5블록 × N + 비교축표 + 통합 차용포인트 |
| METAPHOR | "~처럼", "~에 적용" | 원도메인 5블록 + 타겟도메인 매핑 + 충돌지점 |

상세: `→ references/02_mode_protocols.md`

### ③ 5블록 생성

```
[B1] 정의 + 성공/실패 이유 (3~5줄)
[B2] 구조 (스코프별 블록 변형, 불릿 4~6개)
[B3] WHY — 왜 사람들이 쓰는가·사는가 (2~3줄, 인사이트 관점)
[B4] 차용 포인트 — 우리가 가져올 것 (1~3개)
[B5] 백본 — "{대상} = 한 줄 압축" + 확신도·근거
```

**WHY 블록**은 기능 나열이 아니라 **인간 메커니즘**으로 씀 (예: "지식 신분재", "대체 불가 서사"). 상세: `→ references/03_why_insights.md`

### ④ 백본 수렴

백본 1줄 템플릿: `{대상} = "{핵심 정체성 어구}를 {작동 메커니즘}으로 {결과} 모델"`

예: `HBR = "프리미엄 지식 콘텐츠를 구독+교육+라이선스로 다층 수익화한 B2B+B2C 하이브리드"`

백본 뒤에 **확신도(점수) + 근거 1줄** 필수.

### ⑤ .md 산출

- 파일명: `{대상명}_bench.md` 또는 `{대상명}vs{대상명}_bench.md`
- 섹션 순서: H1(대상명) → 5블록 → 백본 → 확신도
- 템플릿: `→ references/05_output_template.md`
- 옵시디언 볼트 저장 요청 시 wikilink 유지 (`obsidian-markdown` 스킬 협조)

## METAPHOR 모드 (은유 벤치마킹)

**목적:** 원 도메인에서 막혔을 때 타 도메인의 성공패턴을 빌려옴.

```
[원] 스타벅스 = "써드 플레이스 + 리추얼 + 일관성"
       ↓ 매핑
[타] 은행지점을 "업무공간 써드 플레이스 + 방문 리추얼 + 지점 간 일관경험"으로
       ↓ 충돌지점
[반] 금융은 속도·보안 vs 스벅은 체류·감각 → 변형: 디지털은 속도, 오프라인만 체류형
```

은유 카탈로그(28도메인×성공은유): `→ references/04_metaphor_library.md`

## §INV NO_WORK_LABEL (산출물·대화 본질 보호)

| 항목 | 정의 |
|------|------|
| RULE | 산출물·대화 = 인간 언어. 작업 라벨 ZERO. (1만 페이지 1단어 = FAIL) |
| 판정 | "이 단어, 이 대화 밖 사람이 사전 없이 읽을 수 있나?" NO → 작업 라벨 → 금지 |

## 🚨 MUST cascade → paper-engine (절대규칙)

**이 스킬의 모든 리포트·산출물 = paper-engine 허브 MUST 경유.** 단독 md/html 출력 = FAIL.

- **진입점:** 모드별 분석 완료 후, 산출물 생성 직전 paper-engine 발동
- **paper-engine 책임:** MODE_S/M/L · NYT구조 · 3패스삭제 · 4항밀도QC · design-skill cascade 강제
- **DEFAULT_RENDER = 순수 마크다운:** `.md` 산출 시 `<div>`·`<span>`·`style=` 등 인라인 HTML **전면 금지**. 헤더·불릿·표·인용(`>`)·이모지만 허용
- **예외:** 사용자가 `"HTML로"·"박스로"·"벤토로"·"시각화"·"카드로"` 명시시에만 html-div-style·apple-box-design cascade 경유
- **위반 감지:** md 파일에 `<div style>`·`<span style>` 삽입 = 절대규칙 #8 위반 → 재작성

## Gotchas

| 함정 | 대응 |
|---|---|
| 백본 없이 끝냄 | B5 필수. 없으면 "모름" + 왜 수렴 안 되는지 1줄 |
| 구조 블록에 디테일 나열 | 불릿 4~6개 상한. 그 이상은 스포크로 |
| 매출·이용자수 추측 | "모름(30)" 선언. 공개된 범위만 |
| WHY가 기능 나열로 변질 | 인간메커니즘 어구 사용 (신분재·앵커·서사 등) |
| MULTI에서 비교축 없이 나열 | 비교축표 필수. 최소 3축(구조·BM·강점) |
| METAPHOR 매핑 없이 끝냄 | 원→타 매핑 + 충돌지점 둘 다 필수 |
| 길이 초과 | SINGLE은 A4 반쪽. 초과 시 압축 재작성 |
| hedge 섞임 | "~할 수 있다" 발견 시 단정형 or "모름" 치환 |
